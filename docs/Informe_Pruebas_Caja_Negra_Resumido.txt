3.4. INFORME DE PRUEBAS DE CAJA NEGRA

Este informe recoge las pruebas de caja negra realizadas sobre TaskFlow.
A diferencia de las de caja blanca, aquí no se analiza el código por
dentro, sino el comportamiento del sistema desde fuera: lo que ve el
usuario en la aplicación y lo que devuelve la API ante cada petición.

Para no alargar el informe se ha seleccionado un conjunto representativo
de pruebas, centrado en las funciones principales y en las reglas de
negocio más importantes. Se dividen en dos bloques: pruebas funcionales
desde la aplicación de Flutter y pruebas de la API con Postman.


3.4.1. Entorno y metodología

Las pruebas funcionales se ejecutan a mano sobre la aplicación, con una
cuenta de administrador y otra de usuario estándar. Las pruebas de API
se realizan con Postman, enviando peticiones al backend y comprobando el
código de estado y la respuesta. Se parte de un servidor en
funcionamiento y de una base de datos con datos de ejemplo.

Cada prueba indica la acción, el resultado esperado y el obtenido. En
las que requieren evidencia visual se señala dónde insertar la captura.

3.4.4. Conclusión

Las pruebas de caja negra comprueban el comportamiento de TaskFlow desde
fuera, tanto en la aplicación de Flutter como mediante el consumo de la
API con Postman. Los 14 casos seleccionados cubren el inicio de sesión,
la gestión de usuarios, proyectos y tareas, el control de acceso por rol
y las reglas de negocio principales.

Todos los casos se ejecutaron con el resultado esperado. La aplicación
responde de forma coherente ante las acciones correctas y muestra avisos
claros ante las entradas erróneas, y la API devuelve en cada caso el
código de estado adecuado. El resultado confirma que la interacción
entre el frontend, el backend y la base de datos funciona según los
requisitos definidos.
