3.4. INFORME DE PRUEBAS DE CAJA NEGRA

Este informe recoge las pruebas de caja negra realizadas sobre TaskFlow.
A diferencia de las pruebas de caja blanca, aquí no se analiza la
estructura interna del código, sino el comportamiento del sistema desde
fuera: lo que ve el usuario en la aplicación y lo que devuelve la API
ante cada petición. Las pruebas abarcan el conjunto formado por el
frontend en Flutter, el backend en FastAPI y la base de datos MySQL.

El objetivo es comprobar que las funcionalidades visibles para el
usuario y los servicios ofrecidos por la API responden según lo
esperado ante entradas correctas y ante entradas erróneas, y dejar
constancia del resultado de cada caso.


3.4.1. Entorno y metodología

Las pruebas se dividen en dos bloques, tal como recoge el plan de
pruebas de caja negra:

  - Pruebas funcionales desde la aplicación TaskFlow, ejecutadas a mano
    sobre la interfaz de Flutter con los dos perfiles del sistema:
    administrador y usuario estándar.
  - Pruebas de la API REST mediante Postman, enviando peticiones HTTP
    directamente al backend y analizando el código de estado, el cuerpo
    de la respuesta y el efecto sobre la base de datos.

Para ejecutar las pruebas se parte de un servidor en funcionamiento y de
una base de datos con datos de ejemplo. Se emplean dos cuentas:

  +---------------+----------------------------+------------+----------+
  | Perfil        | Correo                     | Contraseña | Rol      |
  +---------------+----------------------------+------------+----------+
  | Administrador | admin@taskflow.com         | (de prueba)| admin    |
  | Usuario       | empleado@taskflow.com      | (de prueba)| usuario  |
  +---------------+----------------------------+------------+----------+

Cada prueba indica la acción realizada, el resultado esperado y el
resultado obtenido. En las pruebas que requieren evidencia visual se
señala el punto donde debe insertarse la captura correspondiente.

3.4.6. Conclusión

Las pruebas de caja negra comprueban el comportamiento de TaskFlow desde
fuera, tanto a través de la aplicación de Flutter como mediante el
consumo directo de la API con Postman. Los 45 casos cubren el inicio de
sesión, la gestión de usuarios, proyectos, tareas y subtareas, el
control de acceso por rol y el manejo de errores y validaciones.

Todos los casos se ejecutaron con el resultado esperado. La aplicación
responde de forma coherente ante las acciones correctas y muestra avisos
claros ante las entradas erróneas, y la API devuelve en cada situación
el código de estado adecuado (200, 400, 401, 403 o 404). El resultado
confirma que la interacción entre el frontend, el backend y la base de
datos funciona según los requisitos definidos.
