1. INTRODUCCIÓN

Este informe recoge las pruebas de caja blanca realizadas sobre el backend de
TaskFlow, la API REST desarrollada con FastAPI y SQLAlchemy en Python. Las
pruebas de caja blanca examinan el código por dentro: conociendo la
implementación, se diseñan casos que recorren los caminos lógicos relevantes de
cada función, en especial los condicionales que afectan a la seguridad, a la
integridad de los datos y a las reglas de negocio.

El objetivo es comprobar que cada validación implementada en el código responde
como se espera tanto ante entradas correctas como ante entradas erróneas, y
dejar constancia del fichero y las líneas concretas donde reside la lógica que
cubre cada prueba.


2. ENTORNO Y METODOLOGÍA

Las pruebas se han automatizado con pytest y el cliente de pruebas de FastAPI
(TestClient), que ejecuta las peticiones HTTP contra la aplicación sin necesidad
de levantar el servidor.

Para no alterar la base de datos MySQL de producción, las pruebas usan una base
de datos SQLite en memoria. La dependencia get_db de la aplicación se sustituye
por una sesión de pruebas, de modo que cada test arranca con un esquema limpio,
inserta sus propios datos y los descarta al terminar. Las contraseñas se generan
con la misma función de hash que utiliza la aplicación real (bcrypt), y los
tokens JWT se obtienen llamando al endpoint de login, igual que haría la
aplicación móvil.

Ficheros de prueba:
  tests/conftest.py        Configuración común, base de datos de prueba y datos base.
  tests/test_auth.py       Inicio de sesión, tokens JWT y recuperación de contraseña.
  tests/test_roles.py      Control de acceso por rol y rutas de administrador.
  tests/test_usuarios.py   Gestión de usuarios y cambio de contraseña.
  tests/test_proyectos.py  Gestión de proyectos y validación de fechas.
  tests/test_tareas.py     Gestión de tareas, asignaciones y fechas límite.
  tests/test_subtareas.py  Gestión de subtareas y control de acceso.

Comando de ejecución:
  pytest -v
  pytest --cov=app --cov-report=term-missing   (para medir cobertura)

4. DETALLE POR CADA PUNTO DEL PLAN DE PRUEBAS

A continuación se describe cada prueba prevista, el fragmento de código que la
implementa (ruta y rango de líneas) y los casos asociados con su resultado.


4.1. Inicio de sesión con credenciales válidas e inválidas

Código responsable: app/routers/auth.py, líneas 15 a 31 (función login).
El endpoint busca al usuario por correo, verifica la contraseña con bcrypt y, si
todo es correcto, genera un token. Si el correo no existe o la contraseña no
coincide, responde con un error 401 y un mensaje genérico que no revela cuál de
los dos datos ha fallado.

  Caso                                          Entrada                      Esperado   Resultado
  --------------------------------------------  ---------------------------  ---------  ---------
  Login con credenciales válidas                admin + contraseña correcta  200        CORRECTO
  Login con correo inexistente                  correo que no existe         401        CORRECTO
  Login con contraseña incorrecta               admin + contraseña errónea   401        CORRECTO


4.2. Generación y validación de tokens JWT

Código responsable: app/utils/security.py, líneas 21 a 32 (funciones
crear_token y verificar_token). El token se firma con la clave secreta y un
tiempo de expiración. La verificación decodifica el token y devuelve su
contenido, o None si la firma es inválida.

  Caso                                          Entrada                      Esperado          Resultado
  --------------------------------------------  ---------------------------  ----------------  ---------
  Crear y validar un token correcto             datos sub y rol              payload con datos CORRECTO
  Validar un token manipulado                   cadena alterada              None              CORRECTO


4.3. Control de acceso según el rol del usuario

Código responsable: app/routers/usuarios.py, líneas 15 a 27 (funciones
get_current_user y solo_admin). get_current_user resuelve el usuario a partir
del token y rechaza los tokens ausentes o inválidos con un 401. solo_admin
comprueba el rol y devuelve un 403 si el usuario no es administrador.

  Caso                                          Entrada                      Esperado   Resultado
  --------------------------------------------  ---------------------------  ---------  ---------
  Acceso sin token a ruta protegida             sin cabecera de autorización 401        CORRECTO
  Acceso con token inválido                     token falso                  401        CORRECTO
  Usuario estándar en ruta de administrador     token de usuario estándar    403        CORRECTO
  Administrador en ruta de administrador        token de administrador       200        CORRECTO


4.4. Validación de rutas exclusivas para administradores

  Caso                                          Entrada                      Esperado   Resultado

Un usuario estándar solo ve y abre las tareas en las que figura asignado, y solo
puede cambiar el estado, no el resto de campos.

  Caso                                          Entrada                      Esperado          Resultado

4.7. Cambio y recuperación de contraseña

Código responsable: cambio de contraseña en app/routers/usuarios.py, líneas 30
a 41 (cambiar_contrasena); recuperación y restablecimiento en
app/routers/auth.py, líneas 34 a 87 (recuperar_contrasena y
resetear_contrasena). El cambio exige la contraseña actual correcta. La
recuperación responde siempre con un mensaje genérico para no revelar si un
correo está registrado, y el restablecimiento rechaza los tokens inválidos o
caducados.

  Caso                                          Entrada                      Esperado   Resultado
  --------------------------------------------  ---------------------------  ---------  ---------
  Cambiar contraseña con la actual correcta     actual correcta              200        CORRECTO
  Cambiar contraseña con la actual incorrecta   actual errónea               400        CORRECTO
  Recuperar con un correo no registrado         correo inexistente           200 (*)    CORRECTO
  Restablecer con un token inválido             token inexistente            400        CORRECTO

  (*) Responde 200 con un mensaje genérico de forma intencionada, para no
      revelar qué correos existen en el sistema.

4.14. Respuestas de error ante datos inválidos, recursos inexistentes o permisos insuficientes

Este punto es transversal y se comprueba a lo largo de todas las áreas
anteriores. La API devuelve códigos coherentes en cada situación: 401 cuando
falta autenticación o el token no es válido, 403 cuando el rol no autoriza la
operación, 404 cuando el recurso solicitado no existe y 400 cuando los datos
incumplen una regla de negocio. Un ejemplo concreto de integridad es la creación
de una tarea con un usuario asignado inexistente (app/routers/tareas.py, líneas
80 a 83): la operación se rechaza con un 404 y, gracias al orden de las
validaciones, la tarea no llega a guardarse en la base de datos.

  Caso                                          Entrada                      Esperado          Resultado
  --------------------------------------------  ---------------------------  ----------------  ---------
  Crear tarea con proyecto inexistente          proyecto que no existe       404               CORRECTO
  Crear tarea con usuario inexistente           usuario que no existe        404, sin guardar  CORRECTO

6. CONCLUSIÓN

Las pruebas de caja blanca recorren los principales caminos lógicos del backend
de TaskFlow. Los 55 casos cubren el inicio de sesión, la firma y validación de
tokens, el control de acceso por rol, la separación entre administradores y
usuarios estándar, las operaciones sobre usuarios, proyectos, tareas y
subtareas, y las reglas de negocio asociadas a las fechas y al estado de las
tareas.

Todos los casos se ejecutaron con el resultado esperado. El comportamiento
ante credenciales erróneas, recursos inexistentes y permisos insuficientes
responde con los códigos de error adecuados, lo que confirma que la capa de
control del backend se comporta de forma coherente frente a las entradas
correctas y a las erróneas.
